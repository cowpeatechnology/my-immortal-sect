# My Immortal Sect（《我的宗门》）- 核心设计方向与技术架构决策

## Context

这是 My Immortal Sect（《我的宗门》），一个面向微信 / 抖音小游戏、V1 以**单人宗门经营 + 云存档**为核心并保留后续多人扩展参考的修仙题材宗门经营游戏。

**技术栈**：
- 客户端：Cocos Creator + TypeScript + Tiled + Spine/DragonBones
- 服务端：**Go + [Hollywood Actor Framework](https://github.com/anthdm/hollywood)**（actor 模型，见 ADR 0007）
- 数据：PostgreSQL（主）+ Redis（缓存）+ 云端存档
- 配置分发：CDN 热更新（绕过 App Store 审核）

**项目状态**：已进入真实客户端切片、协作流程与项目治理同步推进阶段；当前仍属早期研发，但已不再是“尚未动工”状态。所有代码与文档工作都应遵守当前可见角色协作纪律（见第 6 节）。

经过多轮讨论，核心决策收敛如下:

1. **玩法定位**：RimWorld 式"弟子即因果"涌现叙事；宗门本身是玩家情绪锚点
2. **双系统拆分**：因果（宗门编年史 + 命运预告）+ 神降（异界体验副本）
3. **架构关键洞察**：因果和神降共享同一个 **Storylet 引擎**，V1 技术工作量减半
4. **美术取舍**：大头图标 + 情绪状态为主；Spine 只用于 1v1 对话
5. **V1 范围极简**：神降只做 1 个官方世界，纯对话 + 自动战斗，用世界观自洽解释
6. **长期野心**：神降作为 UGC 平台，作为项目第二增长曲线
7. **服务端基础**：采用 Hollywood Actor Framework，以 actor 模型组织所有业务逻辑；Storylet = Actor 是天然同构（见 ADR 0007）

这份文档是这些决策的正式记录，用作开发的 North Star。

## 1. 核心设计原则

### 1.1 情绪锚点在宗门内部

- 玩家最舍不得失去的是**宗门内部的弟子**，不是神降世界的内容
- 神降是"派弟子去异界",不是"玩家亲自去神降世界"
- 所有高情绪事件必须威胁或馈赠玩家已经建立情感连接的对象

### 1.2 弟子即因果契约

- 稀有弟子（比例约 **1:10**）自带多阶段"因果事件链"
- **接纳** = 获得强力资质 / 技能 + 承担 N 次后续危机
- **拒绝** = 通过后续旁白 / 传闻产生间接情绪反馈（不能是"纯无事发生"）
- 因果必须**可见、可导航、可了结**：
  - 弟子面板常驻"因果未了"标签
  - 触发前有征兆（异象 / 传信 / 弟子噩梦）
  - 承担得当有"了结"时刻（渡劫 → 弟子突破 + 宗门获得传承）
- 设计底线：让玩家觉得**"这笔因果值得承担"**，难但不惩罚

### 1.3 宗门是家，不是关卡

- 宗门永续运转，不存在"通关"
- 建造是情绪投资，不是纯数值加成
- 死亡弟子应留下可见痕迹（墓碑 / 遗物 / 牌位）
- 宗门历史应当可回顾（族谱 / 大事记）

### 1.4 东方式涌现叙事：宿命感，不是意外感

- RimWorld 的情绪来自**突发意外**（焦虑 + 未知）
- 本项目的情绪来自**宿命预知**（天机 + 备战 + 干预）
- 因果系统的设计核心是**"玩家能看到即将发生的事情"**——玩家是知命者，不是受害者
- 这是修仙题材区别于西式生存游戏的灵魂

## 2. 系统架构：因果 + 神降 = 同一个 Storylet 引擎

### 2.1 关键洞察

因果和神降从玩家视角看是两个独立系统，但从底层数据结构看是**同一个东西**：

```
Storylet = 触发条件 + 叙事节点 + 分支选择 + 结果 + 可种下的新 storylet
```

- 因果 = 以"宗门 / 弟子状态"为触发的 storylet 集合
- 神降 = 以"玩家主动进入异界"为触发的 storylet 集合

**两个系统共享同一个引擎，用不同的 UI 框架包装。**

这个架构决策让 V1 技术工作量砍掉约 40%，也让两个系统的耦合变成了一个单点——**神降回来产生的"因果种子"就是新的 storylet**。

### 2.2 Storylet 引擎数据结构（概念草案）

```
Storylet {
    ID              // 永不修改
    Category        // karma | divine_descent
    Triggers        // 触发条件数组
    Stages          // 多阶段节点
    Prerequisites   // 先决条件
}

Stage {
    NarrativeText   // 叙事文本
    Choices         // 分支选项
    AutoBattleSpec  // 可选：自动战斗描述
    NextStageDelay  // 下一阶段触发延迟（现实时间）
}

Choice {
    Text
    Requirements    // 例如："灵根 ≥ 中"
    Outcomes        // 资源变化 / 弟子状态变化 / 剧情变量
    SpawnsStorylets // 可种下新的 storylet（因果种子）
}
```

详细 DSL 规范在 M0 前置工作里确定。

### 2.3 因果系统（宗门编年史前端）

- 表现形式：可翻阅的"宗门志"
- 包含**已发生事件**（回忆）+ **即将发生事件**（预告）
- 弟子面板常驻"因果未了"标签
- 触发前有征兆，不静默触发
- 情绪核心：**宿命感**

### 2.4 神降系统（异界体验前端）

#### V1 最小形态（Storylet-Only）

- 只有 **1 个官方神降世界**
- 纯对话框 + 分支选择 + 自动战斗结果
- **世界观自洽解释**："玩家修为不足，只能神降体验对话与观察，无法亲身操作战斗"
- 这既是技术简化，也是合理的叙事分级，**还为将来深度神降留下天然的付费解锁点**

这是一个被称为 **Storylet System**（故事小单元系统）的成熟玩法形态，代表作品：Sunless Sea、Fallen London、Reigns、Choice of Games 系列。内容产出效率极高。

#### 未来深度神降的分层解锁（长期）

| 玩家境界 | 神降深度 |
|---|---|
| 筑基前 | 纯对话 + 自动战斗（V1 范围） |
| 筑基 | 简单交互 + 选择战术 |
| 金丹 | 派弟子 + 实时指挥 |
| 元婴 | 可改变异界命运 |
| 化神 | UGC 编辑，成为世界创造者（长远） |

#### 与因果的联动

神降归来后，弟子身上会被种下新的因果种子，进入因果系统继续发酵。**这是两个系统唯一的耦合点**，也是循环闭环的关键。

## 3. 美术与表现层架构

### 3.1 三层表现分工

| 层级 | 内容场景 | 渲染方式 | 同屏上限 |
|---|---|---|---|
| **主视图** | 大地图宗门经营 | 大头图标（情绪可变） | 30+ |
| **Storylet 视图** | 因果 / 神降叙事 | 立绘 / 大头 + 对话 UI | 1~2 |
| **战斗视图** | 冲突事件 | 大头 + 飘字 + 特效 | 15~20 大头 |

### 3.2 大头图标系统

**必备情绪状态（≥ 8 种）**：
愉悦 / 平静 / 困倦 / 沮丧 / 愤怒 / 恐惧 / 受伤 / 濒死
（扩展：专注修炼 / 顿悟 / 走火入魔）

**组件化差异化层**：
- 脸型底板：5~8 种
- 发型 / 发饰 / 冠冕层：随境界 / 装备动态切换
- 表情覆盖层：8 种状态
- 特殊标记层：因果弟子独占边框 / 光效

组合空间：5 × 8 × 8 ≈ 320 种基础外观。

**铁律：因果弟子必须独占手绘标识**
普通弟子走组件拼装，但因果弟子（约 10%）必须每个都有独特发色 / 独占配饰 / 专属光效。玩家不会记住"发型 3 + 底板 2"，只会记住符号化独特角色。

### 3.3 Spine 使用范围

**只用于**：
- 弟子详情页 1v1 对话互动
- 境界突破 CG 时刻
- 神降事件主角色立绘

**规格**：
- 单场景同屏 Spine = 1
- Cached Animation 模式
- 切角色时销毁旧实例
- 首批只做 10~15 个核心角色

**定位**：Spine 是"仪式性资源",配合弟子养成进度做"对话立绘解锁",本身就是养成反馈闭环。

### 3.4 战斗符号化语言

战斗不靠动画，靠统一的符号系统：
- 伤害飘字（颜色区分暴击 / 属性 / 治疗）
- buff / debuff icon 固定槽位
- 关键技能触发时全屏特效 + 0.3s 停顿
- 濒死：大头变红 + 抖动
- 陨落：变灰 + "陨落"飘字
- 关键事件用半秒级停顿 + 全屏文字打断节奏

**第一版定 20~30 个符号，严格管控扩展。** 做好了是风格，做不好是混乱。

## 4. 技术栈与后端架构

### 4.1 技术栈选型

| 层 | 技术 | 理由 |
|---|---|---|
| 客户端 | Cocos Creator + TypeScript | 小游戏生态成熟 |
| 服务端运行时 | **Go + [Hollywood Actor Framework](https://github.com/anthdm/hollywood)** | Actor 模型天然契合"服务端权威 + Storylet 引擎 + 模块硬边界"三条核心决策（见 ADR 0007） |
| 主数据库 | PostgreSQL | 关系型适合弟子 / 因果事件等结构化数据 |
| 缓存 | Redis | 会话 / 热数据 / 事件队列 |
| 客户端协议 | WebSocket + HTTP + Protobuf | WS 做实时推送 + HTTP 做关键操作；Protobuf 贯穿 |
| 节点间通信 | Hollywood dRPC (集群化后) | 原生支持，Consul/mDNS 服务发现 |
| 配置热更 | CDN + 版本号 | 绕过 App Store 审核发布内容更新 |

### 4.2 服务端架构（Hollywood Actor Model）

**基础框架**：Hollywood Actor Framework（见 ADR 0007 完整决策记录）。所有业务逻辑以 actor 为基本单元，通过监督树（supervision tree）组织故障隔离与状态持久化。

#### 核心原则

- **服务端权威**：所有关键状态由服务端 actor 计算，客户端只做展示和预测
- **Actor mailbox 天然串行化**：对单个实体的操作不存在并发，**业务逻辑中没有机会写出并发 bug**
- **Storylet ≡ Actor**：每个进行中的 storylet = 一个短命 actor，不需要独立的状态机调度器；因果和神降共享这套抽象
- **持久化走 middleware**：自建 `PGStore` 实现 Hollywood `Storer` 接口，挂到 `WithPersistence`
- **客户端协议走 Gateway**：Cocos ↔ Actor 系统由独立 `GatewayActor` 翻译（WebSocket/HTTP + protobuf）
- **Storylet 引擎双端对称**：客户端 TypeScript 预演 = 服务端 Go 权威，两份实现通过 protobuf schema 对齐

#### Actor 层级

```
Hollywood Engine (root)
└── GatewayActor (WebSocket/HTTP 协议翻译层)
    └── PlayerSupervisor (per online player)
        ├── SectActor              (宗门状态)
        ├── DiscipleActor × N      (弟子，每个独立 actor)
        ├── KarmaFrontendActor     (编年史前端状态)
        ├── DescentFrontendActor   (活跃神降会话)
        ├── ActiveStoryletActor × N (进行中的因果/神降，短命)
        └── SyncActor              (增量同步消息聚合)
```

**故障隔离粒度**：
- 玩家离线 → 整棵 PlayerSupervisor 子树 Poison + 状态持久化
- 单个 storylet 崩溃 → 只重启它，其他不受影响
- 单个弟子 bug → 只恢复该 actor，宗门和其他弟子不受影响

#### 模块划分

```
server/
├── cmd/gameserver/main.go       # 入口：初始化 Hollywood Engine + Gateway
├── internal/
│   └── slggame/                 # 我们的业务代码全部放这里（不动 Hollywood 源码）
│       ├── gateway/             # GatewayActor（WS/HTTP 协议翻译）
│       ├── player/              # PlayerSupervisor
│       ├── sect/                # SectActor
│       ├── disciple/            # DiscipleActor
│       ├── storylet/            # Storylet actor 基类 + 注册表（共享引擎）
│       ├── karma/               # 因果专有 actor
│       ├── descent/             # 神降专有 actor
│       ├── sync/                # SyncActor（增量同步）
│       ├── anticheat/           # 服务端权威校验（middleware 或独立 actor）
│       └── storage/
│           └── pgstore.go       # 实现 Hollywood Storer 接口
├── proto/                       # 我们自己的 protobuf（独立，不与 Hollywood 混）
├── configs/                     # 游戏配置（与客户端同源）
└── migrations/
    ├── db/                      # PostgreSQL schema 迁移
    └── config/                  # 配置版本迁移
```

#### Hollywood 使用策略（硬约束）

1. **不修改 Hollywood 上游源码**（修 bug 要 PR 回上游）
2. **定制一律走 Middleware / 自定义 Actor / 自定义 Storer**
3. **业务 protobuf 独立目录** `server/internal/proto/`，不与 Hollywood 的 protos 混放
4. **Gateway 是独立 actor 类型**，不嵌入 Hollywood 内部
5. **fork 只做 bug fix / backport**，不做业务定制
6. **定期 merge 上游**（月度）

详见 ADR 0007。

### 4.3 云存档 + 增量同步协议

#### 核心思想：瘦存档 + 胖配置

- **配置**（事件内容、对白、数值公式）在客户端 + CDN，不通过存档同步
- **存档**只存**状态引用** + **变量快照**（如事件 ID、选择索引、剩余阶段）
- **同步粒度**按字段，不是整档全传
- 一次常规操作的同步负载目标：**几十到几百字节**

#### 存档结构（逻辑层）

```
PlayerProfile
├── SectState           # 宗门基础状态
├── ActiveDisciples[]   # 活跃弟子（全量）
├── DiscipleArchive     # 历史弟子索引（懒加载）
├── ActiveStorylets[]   # 进行中的因果 / 神降
├── StoryletHistory     # 已完成 storylet 索引（分页）
├── ConfigVersion       # 存档创建时的配置版本号
└── SaveSchemaVersion   # 存档协议版本
```

#### 同步时机

- **登录**：拉取核心状态 + 活跃弟子 + 进行中 storylet（总大小目标 < 100KB）
- **常规操作**：单字段增量 PATCH
- **事件触发**：服务端权威决定，推送给客户端
- **断线重连**：按 last_action_timestamp 补发错过的推送

#### 离线时间处理

**决策**：因果事件按**现实时间**推进，不按"玩家在线时间"。这符合"宿命"的设计意图——**天机不等人**。玩家下次登录会看到"离开期间发生了什么"的**弟子日记**界面。

### 4.4 配置表驱动 + 版本迁移

#### 五条铁律（第一版就必须做对）

1. **事件 / 弟子 / 建筑的 ID 永不修改，只能新增** —— 不可破坏
2. **每份存档记录 configVersion**
3. **每次配置变更写迁移脚本**（`migrations/config/v1_to_v2.go`）
4. **CI 必须跑"旧存档 + 新配置"兼容性测试**
5. **删除内容走 deprecation 流程**，先标记再下线

很多手游死于上线后第一次大更新把存档搞崩。这几条加起来约一周工作量，但能救项目的命。

#### 配置表工具链

```
策划 Excel / Google Sheet
    ↓ (导出脚本)
shared/configs/*.json
    ↓ (类型生成)
client/types/config.d.ts    # 客户端 TypeScript 类型
server/configs/*.go          # 服务端 Go struct
    ↓ (打包上传)
CDN
    ↓ (客户端拉取)
游戏运行时
```

## 5. 技术红利

相对最初方案的量化收益汇总：

| 指标 | 原方案 | 本方案 |
|---|---|---|
| 同屏 Spine 峰值 | 30+ | **1** |
| 低端机战斗帧率目标 | ~20 fps | **≥ 45 fps** |
| 因果 + 神降代码量 | 2 套独立引擎 | **1 套共享 Storylet 引擎** |
| 新增神降世界边际成本 | 高（场景 + 美术） | **只写配置表** |
| 新增普通弟子美术成本 | 全身动作 | **组件拼装（接近零）** |
| 常规操作同步带宽 | 整档上传 | **几十到几百字节** |
| 热更新游戏内容 | 需发版 | **CDN 热更** |
| 买量素材制作成本 | 高 | **约 1/5** |

## 6. Codex 协作纪律

**Codex 是本项目主力代码实现者。** 以下纪律是防止项目 3 个月后变成屎山的底线。这部分和其他设计决策同等重要，不能省。

### 6.1 六条纪律

1. **根目录必须有 `AGENTS.md`，项目内必须有 `.codex/` 规则** —— 写仓库级章程、命名规范、禁止事项和角色模板，对抗风格漂移的第一道防线。
2. **TypeScript strict + 强类型配置表** —— 禁止 `any`，所有配置表必须生成类型定义。类型是 Codex 的止血带。
3. **每个模块必须有测试** —— 重点：Storylet 引擎、存档序列化、战斗符号系统、因果触发器。目标不是覆盖率，是"改 A 不会偷偷破坏 B"。
4. **频繁提交，小步快跑** —— 每个通过验收的循环至少要有一次可追踪 commit，大循环允许拆成多个小 commit。
5. **大改动走 Plan 模式** —— 范围超过 1 个模块的改动先 plan 后 code。不要直接说"把因果系统实现一下",要拆成小步骤。
6. **模块边界硬约束** —— 模块间通过事件总线 / 接口通信；每个模块有 `index.ts` 导出公共 API；改动范围在 plan 里明确写"只动 X，不动 Y"。

### 6.2 `AGENTS.md` + 项目级 Codex 文档核心内容清单

- 架构分层图（客户端 / 服务端 / 配置）
- 命名规范（文件名 / 类型名 / 常量名）
- 禁止事项（不要 any、不要越界改文件、不要 console.log 留在代码里）
- Storylet 引擎的数据约定
- 存档格式的不可变字段
- 配置表的 ID 命名规则
- 新增事件 / 弟子 / 建筑的模板路径
- Git commit 规范
- 主管派单模板、worker 回传模板、feature loop 模板

### 6.3 非代码工作（Codex 做不了，必须人完成）

- 美术资源（大头、Spine、图标、特效）
- 音乐音效
- 事件文案（Codex 可出草稿，最终需人审核）
- 真机性能测试（需要真实设备）
- 游戏手感与节奏调优（需要肉眼判断）
- 产品决策

### 6.4 预估周期（一人 + Codex）

| 模块 | 周期 | 备注 |
|---|---|---|
| M0 技术垂直切片 | 3~4 周 | 含 Hollywood 集成 + Gateway + PGStore |
| Hollywood Gateway Actor | 1 周 | WebSocket/HTTP 协议翻译 |
| PGStore（Hollywood Storer 实现） | 1 周 | PostgreSQL 持久化 |
| Storylet 引擎（基于 actor，双端对称） | 2 周 | 服务端 Go + 客户端 TS 预演 |
| 因果 UI + 编年史 | 1 周 | 客户端 |
| 大头渲染 + 情绪切换 | 1 周 | 客户端（不含美术） |
| 宗门建造基础 | 2 周 | 客户端 + SectActor |
| 存档版本迁移机制 | 1 周 | 配置 + Storer 层 |
| 神降 V1 内容填充 | 2~4 周 | 取决于内容量 |

**总计**：约 12~16 周代码工作，**不含美术产出、不含调优、不含内容打磨**。

**相对从零自建 Go runtime 的节省**：Hollywood 省下了 mailbox / 监督树 / 重启策略 / 集群原语等约 4~6 周基础设施工作，这部分时间被重新分配到 Gateway 和 PGStore 上（见 ADR 0007）。

## 7. 风险与缓解

### 风险 1：战斗可读性
- **症状**：飘字 + 特效过多看不清
- **缓解**：限制同时飘字数；关键事件强制停顿；战斗日志常驻侧栏

### 风险 2：弟子辨识度退化
- **症状**：组件化让普通弟子长得像
- **缓解**：因果弟子独占手绘；名字 / 出身 / 命格三栏始终可见

### 风险 3：Spine 质量门槛上移
- **症状**：Spine 成了唯一"高清内容",质量不够反而显粗糙
- **缓解**：少而精；首批 10~15 个；对话立绘做"解锁"仪式感

### 风险 4：配置版本迁移失控
- **症状**：更新后老存档引用已删除事件
- **缓解**：ID 永不修改铁律 + 迁移脚本 + CI 兼容性测试（见 4.4）

### 风险 5：服务端成本与 devops 负担
- **症状**：持续的服务器 / 带宽 / DB 成本吞噬现金流；部署 / 监控 / 备份耗时超出预期
- **缓解**：
  - 初期用云厂商免费额度（阿里云 / 腾讯云 / AWS）
  - 数据库选成熟云托管（RDS）避免自建
  - 静态资源走 CDN 不走动态接口
  - 事件触发计算批量化，不是每次玩家动作都即时算
  - 监控用开箱即用方案（Grafana Cloud / 阿里云监控），不造轮子
  - 设计"日活 → 服务器成本"的单位经济模型

### 风险 6：后端经验缺口
- **症状**：Codex 能写代码，但生产环境 devops 是另一回事
- **缓解**：
  - M0 先用最简单部署（单机 Docker）
  - 避免微服务，单一 Go 二进制部署
  - 先正确，再优化

### 风险 7：内容文案产出
- **症状**：Storylet 引擎能容纳无限内容，但内容本身需要人写
- **缓解**：
  - V1 聚焦 10~15 条核心因果链 + 1 个神降世界，不贪多
  - 建立事件写作模板（格式化 DSL）降低写作门槛
  - 预留 "Codex 辅助写草稿 + 人审核" 的工作流

### 风险 8：UGC 野心过早
- **症状**：V1 就追求 UGC 编辑器 → 空货架 → 玩家流失
- **缓解**：UGC 是第二增长曲线，严格按第 9 节分阶段推进

## 8. 下一步行动

### M0 前置工作（≤ 1 周）

M0 开工前必须完成三件事：

1. **美术最小套件**：1 张符合最终规格的大头底板 + 8 种表情覆盖层 + 图集打包方案
2. **存档协议设计稿**：字段清单 + 同步粒度 + 版本号策略
3. **Storylet DSL 草案**：配置表格式 + 第 1 条完整因果链的文本稿

### M0：技术垂直切片（3~4 周）

**目的**：打通客户端 + Hollywood actor 服务端完整链路，压测技术红线。

**内容**：
- 客户端骨架：Cocos Creator 工程 + 大头渲染组件 + 情绪切换
- 服务端骨架：
  - 引入 fork 的 Hollywood
  - `GatewayActor`（WebSocket + protobuf）
  - `PlayerSupervisor` + `SectActor` + `DiscipleActor` 最小监督树
  - `PGStore` 实现 Hollywood `Storer` 接口
  - 登录 / 同步 / 推送最小链路
- **一条完整 storylet actor 链路**：接纳因果弟子 → 三次后果触发 → 了结
- 配置表工具链：Excel → JSON → 双端类型生成
- 云存档读写 + 增量同步协议原型
- 存档版本迁移机制原型
- `AGENTS.md` + `.codex/` 项目配置 + 测试基础设施

**压测目标**：
- 30 弟子同屏 + 战斗事件时客户端帧率 ≥ 45 fps（红米 Note 8 级）
- 单次常规操作同步带宽 ≤ 200 字节
- 登录后首屏渲染 ≤ 5s
- 服务端单机支撑 ≥ 500 并发，p99 响应 ≤ 200ms

**此步不过则立即回到设计板，不继续堆功能。**

### M1：美术管线验证

- 大头组件系统（底板 / 发型 / 表情 / 装饰 四层）
- 1 个主角 Spine 立绘（5 种表情切换）
- 战斗特效库（20~30 个常用符号）
- 5 个因果弟子独占手绘标识

### M2：Storylet 内容流水线

- 完整的 Storylet DSL 规范
- 事件编辑器 / 配置模板（策划友好）
- 10 条完整主因果链 + 1 个神降世界
- 内容产出速率测量（1 人 × 1 周能产几个 storylet 阶段）

### M3：差异化一句话

- 必须回答：**"凭什么玩家选我而不是《最强祖师》？"**
- 答案必须能直接变成买量素材第一帧
- 答不出来就改立项，不要开发

## 9. 长期规划：第二增长曲线（神降 UGC 平台）

UGC 是本项目的终极野心，但**必须分阶段实现**，避免"空货架"死亡螺旋。

### 阶段路径

```
阶段 1（V1 上线）
  ├─ 1 个官方神降世界（纯对话 + 自动战斗）
  └─ 验证：Storylet 玩法是否成立

阶段 2（V1 + 3 月）
  ├─ 官方持续新增神降世界（每月 1~2 个）
  ├─ 数据驱动：哪种世界留存最好？
  └─ 开放"深度神降"（随玩家境界解锁）

阶段 3（V1 + 6~9 月）
  ├─ 隐藏创作者试水
  ├─ 挑选主播 / KOL 做"定制神降",官方代编辑
  └─ 验证：创作者生态可行性

阶段 4（V1 + 12 月+）
  ├─ 配置式 + 模板式 UGC 编辑器公测
  ├─ 不做脚本式（Roblox 级投入，不现实）
  └─ 审核 + 举报 + 运营机制上线

阶段 5（未来）
  ├─ 创作者分成 + 精选榜 + 订阅制
  └─ 形成正向循环：创作者赚钱 → 更多创作者加入
```

### 关键决策节点

- **阶段 1 → 2**：当官方神降世界日活留存稳定
- **阶段 2 → 3**：当玩家对"希望有更多世界"的呼声明确
- **阶段 3 → 4**：当试水创作者产出的内容留存不低于官方
- **阶段 4 → 5**：当平台日活足以撑起分成池

**原则：UGC 不是 V1 功能，它是项目成功的放大器，不是成功的前提。**

## 11. 验证方式

1. **真机压测**：30 弟子 + 战斗事件，帧率 ≥ 45 fps（红米 Note 8 级）
2. **同步协议压测**：常规操作带宽 ≤ 200 字节
3. **服务端并发压测**：单机 ≥ 500 并发，p99 响应 ≤ 200ms
4. **存档迁移测试**：老配置存档读入新配置不崩溃
5. **盲测情绪连接**：5 位玩家玩 1 小时后能说出 ≥ 2 个弟子的故事
6. **Storylet 可读性**：新手玩家能在 30 秒内看懂一场 storylet 战斗胜负
7. **买量素材测试**：用大头风格产 3 条素材，A/B 测试 CTR
