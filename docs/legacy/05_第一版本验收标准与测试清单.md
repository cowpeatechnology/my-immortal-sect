# 05_第一版本验收标准与测试清单

## 1. 验收目标
V1 验收的目标不是“内容很多”，而是确认：
1. 宗门经营骨架成立
2. 技术路线适合小游戏平台
3. 后续扩展不需要大改底层

## 2. 产品验收标准
### 玩家理解成本
- 新玩家在 3~5 分钟内理解：
  - 建筑能做什么
  - 弟子为什么要分工
  - 资源如何产出与消耗
  - 神降为什么值得触发

### 成长反馈
- 玩家能明显感受到：
  - 建筑升级后效率提高
  - 弟子突破后效率提升
  - 资源循环越来越顺
  - 神降结果会影响宗门发展

### 修仙味
V1 至少要让玩家感受到：
- 有灵根/资质差异
- 有境界成长
- 有炼丹/炼器/法宝等修仙标签
- 有宗门在“修仙”而不是“普通农场”

## 3. 功能验收标准
### 地图
- Tiled 地图成功加载
- 点击格子准确
- 等距坐标与逻辑坐标准确映射
- 建筑占地检查正确

### 建筑
- 至少 6 个建筑可正常建造
- 建筑可升级
- 升级后参数生效
- 建筑状态可正确显示

### 资源
- 至少 6 种资源有统一定义
- 可采集 / 入库 / 消耗 / 产出
- 不出现负库存、重复产出等重大错误

### 弟子
- 弟子可生成
- 弟子可接任务
- 弟子可移动并完成任务
- 弟子可修炼并突破至少 2 个阶段

### 工作与 Reservation
- 不会出现多个弟子同时抢同一资源导致逻辑错误
- 任务失败可回滚或重派
- 缺料时建筑能正确停工

### 神降
- 至少 1 套神降事件模板可运行
- 事件选项可生效
- 奖励能正确回流宗门

### 存档
- 退出后重新进入，宗门核心状态可恢复
- 建筑、资源、弟子、境界数据不丢失

## 4. 技术验收标准
- simulation core 与 Cocos Node/界面分离
- 建筑/资源/境界主要依赖配置
- 新增一种建筑不需要大改底层
- 微信小游戏真机可运行
- 主要场景不卡死、不频繁崩溃
- 主包只保留核心内容
- 神降与非核心素材能后置加载

## 5. 数值验收标准
- 玩家能通过基础建筑形成正向产出
- 升级不是摆设
- 资源紧缺但不至于卡死
- 突破有明确门槛
- 突破后有明确收益
- 神降收益有吸引力但不会碾压宗门内经营收益

## 6. 试玩验收问题单
1. 你能理解宗门里每种建筑的作用吗？
2. 你能理解弟子为什么去干某个工作吗？
3. 你是否会主动想升级某些建筑？
4. 你是否觉得资源链条是合理的？
5. 你是否愿意为了神降奖励继续玩？
6. 你是否能感受到修仙题材，而不是普通经营？
7. 哪个系统最不清楚？
8. 哪个步骤最卡、最烦、最无聊？

## 7. 不通过的典型情况
- 建筑和资源循环不成立
- 弟子工作大量异常
- 神降与宗门循环割裂
- 存档频繁损坏
- 新增建筑必须改大量底层代码
- 真机表现明显不可接受

## 8. 通过后的下一步
如果 V1 验收通过，优先进入 V1.1：
1. 增加建筑与资源链
2. 增加神降事件模板
3. 增加法宝 / 炼器表现
4. 增加灵兽/养殖轻量系统
5. 做更好的新手引导与留存内容
